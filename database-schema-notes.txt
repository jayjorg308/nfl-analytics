═══════════════════════════════════════════════════════════════
NFL ANALYTICS — SCHEMA v1 (FINAL, POST-PARQUET-SPIKE)
═══════════════════════════════════════════════════════════════

This document captures the v1 schema as designed through extensive
design conversations, an ADR review cycle, and a parquet-in-Node
spike that confirmed the ingestion architecture. For architectural
context behind specific decisions, see docs/adr/.

──────────────────────────────────────────────────────────────
REFERENCE / STATIC DATA
──────────────────────────────────────────────────────────────

TABLE: season
  - id (PK)
  - year                        e.g., 2026
  - startDate                   first regular season game
  - endDate                     Super Bowl date
  - currentWeek                 cached for performance
  - isComplete                  boolean

TABLE: stadium
  - id (PK)
  - name                        e.g., "Allegiant Stadium"
  - city
  - stateOrCountry              "NV" or "United Kingdom"
  - latitude
  - longitude
  - capacity
  - isDome                      boolean
  - hasRetractableRoof          boolean
  - surfaceType                 "grass" / "field turf" / "monofilament"
  - elevationFeet               relevant for Denver, Mexico City
  - isPrimaryHomeStadium        boolean (false for neutral-site-only venues)
  - homeTeamId                  FK -> team (nullable for neutral-site-only)

TABLE: team
  - id (PK)
  - name                        e.g., "Seattle Seahawks"
  - abbreviation                e.g., "SEA"
  - conference                  AFC / NFC
  - division                    e.g., "NFC West"
  - homeStadiumId               FK -> stadium
  - logoUrl
  - primaryColor                hex e.g., "#002244"
  - secondaryColor              hex e.g., "#69BE28"

TABLE: player
  - id (PK)
  - name
  - position                    QB / RB / WR / TE / etc.
  - jerseyNumber
  - heightInches
  - weightLbs
  - dateOfBirth
  - rookieYear
  - headshotUrl

TABLE: playerTeamMembership
  - id (PK)
  - playerId                    FK -> player
  - teamId                      FK -> team
  - startDate
  - endDate                     nullable (null = current)
  -- Handles mid-season trades. A player has one active membership
  -- at a time but multiple over a career.

──────────────────────────────────────────────────────────────
GAMES AND RESULTS
──────────────────────────────────────────────────────────────

TABLE: game
  - id (PK)
  - seasonId                    FK -> season
  - week                        1-18 regular season, plus playoff weeks
  - gameType                    regular / wildcard / divisional / conference / superbowl
  - homeTeamId                  FK -> team
  - awayTeamId                  FK -> team
  - stadiumId                   FK -> stadium
  - gameDateTime
  - isNeutralSite               boolean
  - isInternational             boolean
  - homeScore                   nullable until played
  - awayScore                   nullable until played
  - status                      scheduled / in-progress / final
  -- Weather (relevant for outdoor games; null for dome games):
  - temperature                 nullable
  - windMph                     nullable
  - precipitationChance         nullable
  - weatherCondition            e.g., "clear" / "snow" / "rain"
  -- Note: in-progress live score is NOT stored. Live-score badge
  -- fetches from ESPN's hidden API on demand (see ADR-0006).

──────────────────────────────────────────────────────────────
DRIVES (NEW IN POST-SPIKE REVISION)
──────────────────────────────────────────────────────────────

TABLE: drive
  - id (PK)
  - gameId                      FK -> game
  - driveNumber                 the canonical drive number (from nflverse fixed_drive)
  - possessionTeamId            FK -> team
  - playCount                   number of plays in the drive
  - timeOfPossession            string format "4:30" (preserves the colon format)
  - firstDowns                  count of first downs achieved on this drive
  - insideTwenty                boolean; did the drive reach the opponent's 20
  - endedWithScore              boolean
  - quarterStart                quarter when drive began
  - quarterEnd                  quarter when drive ended
  - yardsPenalized              total penalty yards on the drive
  - startTransition             e.g., "KICKOFF" / "PUNT" / "INTERCEPTION" / "FUMBLE"
  - endTransition               e.g., "TOUCHDOWN" / "PUNT" / "FIELD_GOAL" / "TURNOVER"
  - gameClockStart              string e.g., "04:58"
  - gameClockEnd                string e.g., "00:28"
  - startYardLine               string e.g., "ARI 29"
  - endYardLine                 string e.g., "NO 4"
  - result                      e.g., "Touchdown" / "Punt" / "Field goal" / "Turnover"
  -- One row per drive per game. Unique constraint: (gameId, driveNumber).
  -- Drive data is replicated on every play row in the parquet; ingestion
  -- deduplicates to one row per (gameId, driveNumber).

──────────────────────────────────────────────────────────────
PLAY-BY-PLAY (THE FOUNDATION OF EPA AND ADVANCED METRICS)
──────────────────────────────────────────────────────────────

TABLE: play
  - id (PK)
  - gameId                      FK -> game
  - driveId                     FK -> drive (NEW — replaces raw drive number)
  - quarter                     1-4, 5+ for OT
  - timeRemainingSeconds        seconds remaining in quarter
  - gameSecondsElapsed          for cross-quarter calculations
  - possessionTeamId            FK -> team (offense)
  - defensiveTeamId             FK -> team (defense)
  - down                        1-4, nullable for special situations
  - distance                    yards to first down
  - yardlineSide                "own" / "opp" / "50"
  - yardlineNumber              0-50
  - yardline100                 0-100, normalized (0 = own endzone, 100 = opp endzone)
  -- Score state before this play (from posteam_score / defteam_score in parquet):
  - scoreOffense
  - scoreDefense
  -- Play type and outcome:
  - playType                    pass / run / punt / field_goal / kickoff / etc.
  - yardsGained                 nullable
  - isTouchdown                 boolean
  - isFieldGoal                 boolean
  - isFieldGoalSuccessful       boolean
  - isPunt                      boolean
  - isTurnover                  boolean
  - turnoverType                interception / fumble / nullable
  - isPenalty                   boolean
  - penaltyYards                nullable
  -- Players involved:
  - passerId                    FK -> player, nullable
  - rusherId                    FK -> player, nullable
  - receiverId                  FK -> player, nullable
  - tacklerId                   FK -> player, nullable (could expand to multi-player table later)
  -- EPA values (from nflfastR, ingested as-is in v1):
  - expectedPointsBefore        EP at start of play (parquet: ep)
  - expectedPointsAfter         EP at end of play
  - epa                         the change (epAfter - epBefore)
  - isSuccessful                boolean (RENAMED from successRate; from parquet: success)
  - isExplosive                 boolean, 20+ yard pass or 10+ yard run
  -- Advanced analytical fields (NEW — added after parquet spike):
  - cpoe                        completion percentage over expected, nullable for non-pass plays
  - xpass                       expected pass rate at this game situation
  - passOverExpected            actual pass indicator minus xpass (parquet: pass_oe)
  -- Pass-specific:
  - passLength                  short / deep / nullable
  - airYards                    nullable
  - yardsAfterCatch             nullable
  - isComplete                  nullable boolean
  -- Run-specific:
  - rushDirection               left / middle / right / nullable

──────────────────────────────────────────────────────────────
PLAYER PERFORMANCE
──────────────────────────────────────────────────────────────

TABLE: playerGame
  - id (PK)
  - playerId                    FK -> player
  - gameId                      FK -> game
  - teamId                      FK -> team (which team they played for)
  - opponentTeamId              FK -> team
  - isHome                      boolean
  - snapsPlayed
  - snapShare                   decimal 0-1
  -- Passing stats (primary for QB, nullable for others):
  - passAttempts
  - passCompletions
  - passYards
  - passTds
  - passInts
  - sacks
  -- Rushing stats:
  - rushAttempts
  - rushYards
  - rushTds
  -- Receiving stats:
  - targets
  - receptions
  - receivingYards
  - receivingTds
  -- Denormalized share metrics (computed at ingestion — see ADR-0011):
  - targetShare                 decimal 0-1; player's targets / team's total targets in game
  - rushAttemptShare            decimal 0-1; player's rushes / team's total rushes in game
  - airYardsShare               decimal 0-1; player's air yards / team's total pass air yards in game
  -- Denormalized opponent context (computed at ingestion, after teamWeekStats):
  - opponentDefenseRankPass     opponent's pass defense rank at time of game
  - opponentDefenseRankRush     opponent's rush defense rank at time of game
  -- Running season-to-date totals (computed at ingestion — see ADR-0011):
  - seasonToDatePassYards       sum of passYards for prior games in season
  - seasonToDateRushYards       sum of rushYards for prior games in season
  - seasonToDateRecYards        sum of receivingYards for prior games in season

TABLE: playerSeasonStats
  - id (PK)
  - playerId                    FK -> player
  - seasonId                    FK -> season
  - teamId                      FK -> team
  - gamesPlayed
  -- Passing aggregates:
  - passYardsTotal, passYardsPerGame
  - passTdsTotal
  - passIntsTotal
  - completionPercentage
  -- Rushing aggregates:
  - rushYardsTotal, rushYardsPerGame
  - rushTdsTotal
  - rushAttemptsTotal
  - yardsPerCarry
  -- Receiving aggregates:
  - receivingYardsTotal, receivingYardsPerGame
  - receivingTdsTotal
  - targetsTotal, targetsPerGame
  - receptionsTotal, receptionsPerGame
  -- Pre-aggregated for query performance; updated after each game

──────────────────────────────────────────────────────────────
TEAM-LEVEL ROLLING STATS (THE SNAPSHOT TABLE)
──────────────────────────────────────────────────────────────

TABLE: teamWeekStats
  - id (PK)
  - teamId                      FK -> team
  - seasonId                    FK -> season
  - week
  -- Overall EPA (defensive values stored as "what they allow":
  -- negative = good defense, positive = bad defense — see ADR-0002):
  - overallEpaPerPlay
  - offensiveEpaPerPlay
  - defensiveEpaPerPlay
  -- Phase-specific EPA:
  - offensivePassEpaPerPlay
  - offensiveRushEpaPerPlay
  - defensivePassEpaPerPlay
  - defensiveRushEpaPerPlay
  -- ELO (computed in-house — see ADR-0004):
  - eloRating                   current rating after this week's games
  - eloChange                   change from previous week
  - eloRankInLeague             optional, derived from rating
  -- Strength of schedule:
  - sosRank
  -- Record:
  - recordWins
  - recordLosses
  - recordTies
  -- Traditional offensive stats:
  - pointsScoredPerGame
  - passYardsPerGame
  - rushYardsPerGame
  -- Traditional defensive stats:
  - pointsAllowedPerGame
  - passYardsAllowedPerGame
  - rushYardsAllowedPerGame
  -- This is a snapshot per team per week. Querying historical
  -- "what was SEA ranked in week 12" reads from this table.

PLAYER PROPS
──────────────────────────────────────────────────────────────

TABLE: prop
  - id (PK)
  - playerId                    FK -> player
  - gameId                      FK -> game
  - propType                    PASS_YDS / RUSH_YDS / REC_YDS / RECEPTIONS /
                                RUSH_ATT / PASS_TDS / PASS_COMPLETIONS /

  - id (PK)

  - line                        e.g., 93.5
  - overPrice                   e.g., -110
  - underPrice                  e.g., -110
  - sourceBook                  e.g., "fanduel"

──────────────────────────────────────────────────────────────
GAME ODDS
──────────────────────────────────────────────────────────────

TABLE: gameOdds
  - id (PK)
  - gameId                      FK -> game
  - marketType                  SPREAD / MONEYLINE / TOTAL
  - actualResult                nullable; populated after game
                                -- SPREAD: actual margin (positive = home covered)
                                -- MONEYLINE: winning teamId
                                -- TOTAL: actual combined score

TABLE: gameOddsSnapshot
  - id (PK)
  - gameOddsId                  FK -> gameOdds
  - capturedAt                  timestamp
  - sourceBook                  e.g., "fanduel"
  -- Spread fields (populated when marketType = SPREAD):
  - homeSpread                  e.g., -4.5
  - homeSpreadPrice             e.g., -112
  - awaySpreadPrice             e.g., -108
  -- Moneyline fields (populated when marketType = MONEYLINE):
  - homeMoneyline               e.g., -230
  - awayMoneyline               e.g., +190
  -- Total fields (populated when marketType = TOTAL):
  - total                       e.g., 45.5
  - overPrice                   e.g., -108
  - underPrice                  e.g., -112

INGESTION INFRASTRUCTURE
──────────────────────────────────────────────────────────────

TABLE: jobQueue
  - id (PK)

AUTH (HANDLED OUTSIDE POSTGRES — see ADR-0005)
═══════════════════════════════════════════════════════════════

User identity is handled by Clerk, not stored in Postgres.

Three access tiers:
  - public read: no auth required (research section, about pages)
  - friend-gated read: Clerk-authenticated user (dashboard, game/player/
    props/team pages)
  - admin write: Clerk-authenticated AND user ID matches hardcoded admin
    (manual data entry/correction UI)

No `user` table in v1. Per-user features (preferences, pick tracking)
are explicitly v2.

═══════════════════════════════════════════════════════════════
KEY DESIGN PRINCIPLES BAKED IN
═══════════════════════════════════════════════════════════════

1. TEMPORAL CORRECTNESS
   - teamWeekStats snapshots every week (no "current value only")
   - ELO rating snapshotted alongside other weekly metrics
   - propLineSnapshot and gameOddsSnapshot for line history
   - playerInjuryStatus tracks status changes over time
   - opponentDefenseRank* denormalized onto playerGame at ingestion
   - seasonToDate* totals materialized at ingestion to avoid window-function bugs

2. INDEPENDENT ENTITIES IN SEPARATE TABLES
   - season, stadium, team, player are independent reference data
   - Stadiums separate from teams (handles neutral sites, relocations)
   - Player team membership separate from player (handles trades)
   - Drives separate from plays (enables drive-level analytical queries
     without forcing joins or denormalization)
   - Injuries separate from players (handles multiple injuries per season)
   - Injury status separate from injury (tracks weekly updates)

3. WRONG-SHAPE vs JUST-SLOW (see ADR-0009)
   - Materialize when natural code organization would lead to N+1 queries
     or repeated complex joins (wrong-shape)
   - Query live when single query is just slower than ideal (just-slow)
   - Example wrong-shape: target share used in many views and split queries
     → materialized on playerGame
   - Example just-slow: streak computation for dashboard cards
     → query-time, indexed scan, cheap

4. DENORMALIZED WHERE QUERY PATTERNS DEMAND IT
   - Opponent defense ranks on playerGame (for fast splits queries)
   - Share metrics (target/rush/airYards) on playerGame (used everywhere)
   - Weather on game (always needed alongside game data)
   - Season aggregates pre-computed in playerSeasonStats
   - Running season-to-date totals on playerGame (avoids window functions)

5. STRUCTURAL SYMMETRY
   - Props and game odds follow the same pattern:
     parent record + snapshot table for line history
   - Makes ingestion and UI code consistent

6. PLAY-BY-PLAY AS FOUNDATION
   - Raw play data stored in postgres for direct querying
   - Aggregates pre-computed for UI performance
   - Research questions answerable via SQL against your own DB

7. EDGE COMPUTATION SIGN CONVENTION (see ADR-0002)
   - Defensive EPA stored as "what they allow":
     negative = good defense, positive = bad defense
   - Matchup edge formula: offensiveEpaPerPlay + defensiveEpaPerPlay
     (addition, not subtraction — defensive sign already encodes direction)
   - Positive edge favors offense; negative favors defense
   - Top-edge selection: max absolute value across four matchups

═══════════════════════════════════════════════════════════════
INGESTION ORDER (CHICKEN-AND-EGG MANAGEMENT)
═══════════════════════════════════════════════════════════════

Weekly ingestion sequence after games complete (triggered Sunday/Monday/
Thursday nights ~30min after final game ends — see ADR-0006):

1. Update game records with final scores
2. Process play-by-play parquet:
   a. Extract unique drives per game and insert/upsert into `drive` table
   b. Insert plays into `play` table with driveId FK references
   -- Drive data is replicated on every play row in the parquet, so
   -- deduplicate by (gameId, driveNumber) before drive inserts.
3. Compute and update teamWeekStats:
   - EPA aggregates from plays
   - ELO recalculation (depends on prior week's ELO + this week's results)
   - SOS update
   - Traditional stat aggregates
4. Update playerGame records:
   - Per-player per-game stats from box scores
   - Compute targetShare, rushAttemptShare, airYardsShare from team totals
   - Denormalize opponentDefenseRank* from teamWeekStats (uses values
     computed in step 3 — must happen after)
   - Compute and store seasonToDate* totals (sum prior games in season)
5. Update playerSeasonStats:
   - Roll up player game stats into season aggregates
6. Update injury reports
7. Update prop/odds actualValue and actualResult fields for completed games

Continuous (multiple times per week):

- Snapshot prop lines from The Odds API
- Snapshot game odds from The Odds API
- Update upcoming game weather forecasts (Open-Meteo)
- Update injury status reports

On-demand (page render):

- Live-score badge: ESPN hidden API for in-progress games, cached 30s

Manual (admin write UI):

- Corrections to any table where automated ingestion has gaps
- Particularly injuries (nflverse load_injuries reportedly less reliable)
- Weather post-game corrections for outdoor games

One-time/rare (local Python scripts, not deployed):

- Historical backfill (run from laptop, writes directly to Neon)
- See scripts/backfill/ for nfl_data_py-based scripts

═══════════════════════════════════════════════════════════════
PARQUET FIELD MAPPING NOTES (from spike)
═══════════════════════════════════════════════════════════════

Notes from the parquet-in-Node spike (using hyparquet) confirming
nflverse field mappings and quirks:

PARQUET TYPE QUIRKS:
- Dates and times are BYTE_ARRAY (STRING) — need parsing during ingestion:
  - game_date format: "2025-09-07"
  - time format: "02:00" (MM:SS in quarter)
  - time_of_day format: ISO timestamp "2025-09-07T18:12:20.900Z"
- Boolean-like flags stored as DOUBLE (0.0 / 1.0):
  - pass, rush, complete_pass, interception, success, etc.
  - Convert to actual booleans in TypeScript ingestion code
- Weather is partially denormalized in source data:
  - `weather` (STRING): free-text e.g. "Sunny Temp: 83° F, Humidity: 72%, Wind: NE 9 mph"
  - `temp` (INT32): structured, nullable (always null for domes)
  - `wind` (INT32): structured, nullable (always null for domes)
  - Use the structured fields when populated; ignore free-text version
- Score state distinction:
  - `home_score` / `away_score`: FINAL score, replicated on every play row
  - `total_home_score` / `total_away_score`: in-game score at time of play
  - `posteam_score` / `defteam_score`: in-game score from possession-team perspective
  - Use posteam_score / defteam_score for play.scoreOffense / scoreDefense

KEY PARQUET → SCHEMA MAPPINGS:
- parquet `success` (DOUBLE 0/1) → play.isSuccessful (boolean)
- parquet `cpoe` → play.cpoe
- parquet `xpass` → play.xpass
- parquet `pass_oe` → play.passOverExpected
- parquet `fixed_drive` → drive.driveNumber (use fixed_drive, not raw drive)
- parquet `fixed_drive_result` → drive.result
- parquet `drive_play_count` → drive.playCount
- parquet `drive_time_of_possession` → drive.timeOfPossession
- parquet `drive_first_downs` → drive.firstDowns
- parquet `drive_inside20` (0/1) → drive.insideTwenty (boolean)
- parquet `drive_ended_with_score` (0/1) → drive.endedWithScore (boolean)

v1 SHIP CRITERIA (see ADR-0012)
═══════════════════════════════════════════════════════════════

v1 is "live" when:

1. All 6 gated views feature-complete with real current-season data:
   - Slate Dashboard
   - Game Detail Page
   - Player Page (position-aware: QB / RB / WR-TE variants)
   - Props Page
   - Team Page
   - Team Leaderboard

2. At least one substantive published investigation in the research
   section demonstrating MDX pipeline + interactive charts end-to-end

3. Weekly post-game ingestion stable for 3-4 consecutive weeks without
   manual intervention, including observed recovery from at least one
   minor failure

4. Analytical verification: at least 3 games' worth of matchup edges and
   ELO outcomes hand-verified against the system's outputs

5. Clerk auth working with all three tiers (public / friend-gated / admin)
   enforced correctly

6. Deployed under portfolio domain, public surfaces indexable, gated
   surfaces requiring auth
