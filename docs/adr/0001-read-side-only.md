# Read-side only: no user-generated content in the database

The database stores only facts (games, plays, lines, injuries) and derivations (EPA, ELO, season aggregates); it does not store user-generated artifacts such as picks, notes, predictions, or watchlists. The social and decision-tracking layer already lives in tools the friends use (sportsbooks, spreadsheets, group chat), so replicating it in-DB adds schema complexity without payoff. Considered a "thin journal" posture (lightweight notes/picks ledger keyed to games/props) and a "full research system of record" posture; both rejected at v1 because they require committing to abstractions before the research workflow is well-understood — revisit when a concrete need surfaces (e.g. backtesting personal picks against actuals). The schema's deferral of `gamePrediction` to v2 is the same call applied to model outputs.
