# v1 ingestion runtime and Python boundary

Production ingestion runs in **Vercel cron functions**, sharing a deployment with the Next.js app. Single deployment surface means one auth boundary, one observability story, one log stream — and manual admin overrides (see ADR-0003) coordinate cleanly with cron-driven writes through the same Postgres connection without cross-service transactional concerns. Considered GitHub Actions, separate workers (Railway / Fly.io / Render), and managed workflow services (Inngest / Trigger.dev) and rejected them all on the principle that additional production runtimes compound operational cost for a single-operator personal project. Pro tier is required for the ~15-minute odds-snapshot cadence during active game windows; weekly post-game ingestion and mid-week paths fit comfortably within hobby tier limits.

The TypeScript-Python ecosystem gap is bridged by **reading nflverse parquet releases directly in Node** (`apache-arrow` / `parquetjs`), eliminating Python from the production runtime. nflverse data is just parquet files in GitHub releases — the Node stack pulls them over HTTP, parses, and upserts to Neon Postgres. A second production language runtime carries compounding costs (two dependency trees, two security update cadences, two debugging environments, two sets of operational knowledge) that a single-operator project cannot absorb. Contingency: if a parquet-in-Node prototype reveals significant friction with nflverse's schema (column-type issues, encoding edge cases, performance problems), the fallback is a GitHub Action that runs Python for ingestion while the production app stays pure TypeScript — Action becomes the bridge. Documented as the escape hatch, not the plan.

Heavy weekly jobs that may exceed Vercel's 300s timeout (large play-by-play parquet upserts plus downstream EPA/ELO aggregation) are **chunked via a `jobQueue` table in Postgres** — pending work items tracked as rows, each cron invocation drains as many as fit in its window and marks them complete, the next invocation resumes. Standard pattern, no separate service required, and crash-safe by construction (a job interrupted mid-window picks up on the next tick).

Historical backfill (one-time bulk load of multiple seasons) runs as a **local one-shot Python script** in `scripts/backfill/`, executed from the author's laptop and writing directly to Neon over a connection string. Not deployed, not scheduled, not part of the production system. Uses `nfl_data_py` because Python's nflverse tooling is materially more mature for bulk operations than hand-fetching parquet files in Node — and backfill is rare enough (run once per historical range extension) that pulling it into production would burden the architecture for an operation that doesn't justify it. Idempotent so re-runs don't duplicate data. Lives in the repo for documentation; never deployed.
