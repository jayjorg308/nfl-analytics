# Three-tier auth model: public / friend-gated / admin

v1 splits the site into three access tiers. **Public read** covers the research section (`/research/*`), methodology pages, the about page, and other static content — the portfolio surfaces, which should be aggressively indexable, linkable, and shareable because the project's portfolio value lives in published analytical work. **Friend-gated read** covers the slate dashboard, game/player/team pages, and the props page — anything that surfaces live sportsbook lines or current-week prop data. Gating these is driven by both data-sensitivity concerns (publicly broadcasting current sportsbook lines invites TOS friction with The Odds API and sportsbooks, plus reputational exposure as a third-party line aggregator) and the "anyone with the URL" problem that a two-tier public/admin model can't solve. **Admin write** (`/admin/*`) gates the manual data entry / correction UI from ADR-0003 and ingestion triggers to the author's identity.

Auth is provided by **Clerk**. Free tier comfortably covers a small invite-list audience, integrates with Next.js App Router middleware, and removes the need for a `user` table in Postgres v1 — identity lives in Clerk's managed user store and the application checks Clerk user IDs at gate points (admin route guard, friend-gated middleware). Considered NextAuth with a one-row admin store and rejected as more code for less polish. Considered an env-var-based password for "single admin write" and rejected once the three-tier model required real user identity for the gated tier, not just a write-gate. No per-user features in v1 (preferences live in localStorage; no pick tracking, no favourite teams) — those are explicit v2 conversations, and the Clerk identity layer is positioned to support them without re-architecture.
