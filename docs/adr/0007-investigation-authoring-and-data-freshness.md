# Investigation authoring: MDX + hybrid data freshness

Investigations are authored as `.mdx` files in `content/research/` with React components for interactive charts. The author writes code professionally and git is the natural authoring loop, so CMS dependencies (Sanity, Contentful, Payload) were rejected as vendor lock-in without enough authoring-UX payoff for a single-author site, and a database-backed `post`-table model was rejected as more engineering for less polish. The portfolio differentiator is *interactive React components embedded in published analysis* — readers can hover charts, filter data, toggle methodologies — and MDX is the cleanest path to that pattern. If a guest contributor ever wants to publish, the workflow is "draft in Google Docs, author ports to MDX," not "adopt a CMS."

Chart data freshness is **hybrid**, decided per investigation via a `freshnessMode` frontmatter field. Three investigation subtypes map to three modes:

- `freshnessMode: snapshot` — historical research investigations (e.g. "is L3-vs-season actually predictive?"). Chart data is materialised at publish time to JSON files committed alongside the MDX (`content/research/{slug}/data/{chart-id}.json`). Findings stay aligned with text forever, and renaming a column doesn't break a year-old piece. Default for analyses through a specific point in time.
- `freshnessMode: live` — working dashboard investigations (e.g. "this season's EPA leaders"). Chart components fetch from API routes that query Postgres with appropriate cache/revalidation. Text describes what the chart shows, not specific values, so it doesn't drift as data updates.
- `freshnessMode: hybrid` — mixed pieces. Individual chart components opt in via prop (`<EpaChart mode="snapshot" />` vs `<EpaChart mode="live" />`). Authors follow the "as of [date]" / "currently" text convention to keep narrative numbers aligned with their chart sources.

The hybrid approach resolves the SEO/trust risk of pure live-querying (article text drifting from chart data) without imposing the staleness of pure snapshotting (working dashboards never updating). Frontmatter makes the choice explicit per investigation, so the rendering layer knows which path to take and the author makes a deliberate call rather than an implicit one. Snapshot generation is a small publish-time script that runs the chart components' declared queries against current Postgres and writes JSON; live mode is normal Next.js fetch with revalidation tuned per chart.
