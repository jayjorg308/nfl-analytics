# Splits computation: query-time by default, materialise only when wrong-shape

Splits (Season / L3 / L5 / Home / Away / vs top-10 defences / indoor-outdoor / etc.) are computed at query time by default against `playerGame` and `play`. Pre-aggregation only happens when the natural code organisation would otherwise produce a *wrong-shape* query — N+1 lookups, repeated complex joins, dashboard payloads requiring many parallel per-card queries — not when a single split query is merely slow. This distinction is the operating principle: **wrong-shape gets materialised; just-slow does not**. Just-slow is almost always fine for personal-scale traffic on small data; wrong-shape produces queries that are hard to write correctly the first time and easy to write incorrectly the second time, so they get solved once at the data layer.

Data volume is small (~500 active players × 18 weeks × low-double-digit games per player → well under a million rows in `playerGame`); indexed Postgres on Neon handles arbitrary split queries in single-digit milliseconds. Pre-materialisation would lock in a split vocabulary at exactly the moment when research investigations are expected to introduce new splits ("performance after a loss," "division games on the road," "first-half vs second-half splits"). Keeping splits query-time keeps the system flexible as analytical questions evolve. `teamWeekStats` and `playerSeasonStats` are already pre-materialised — but only because they're read on essentially every page, which makes them pass the wrong-shape test by virtue of frequency.

Two specific materialisations land in v1 because they fail the wrong-shape test:

- **`playerGame.opponentDefenseRankPass` and `opponentDefenseRankRush`** (already present in the schema). Rank-aware splits like "vs top-10 pass defences" would otherwise require a self-join against `teamWeekStats` to find the top-10 set as-of each game, then filter `playerGame`. Denormalising the rank-at-time-of-game onto `playerGame` collapses the query to a single indexed filter. This is **the mechanism** for all rank-based splits — new rank-based splits should follow the same pattern of denormalising rank context onto the per-game row at ingestion, not computing ranks at read time.
- **`weekSummary` SQL view** for the Slate Dashboard. The dashboard renders 13 cards × 4 matchup edges × triple-split per page load; the wrong-shape risk is N parallel per-card queries. The view exposes one row per game for a given week with all card columns (both teams' EPA across phases, records, SOS, weather, most-recent line snapshot, computed top edge, edge label). The dashboard route runs **one query** against `weekSummary`. Starts as a regular SQL view (live computation); promotes to a materialised view only if live computation slows in practice. AI-assisted code that renders dashboard cards via parallel queries should be flagged as wrong-shape and refactored to use the view.

No app-level cache (Redis, Vercel KV) in v1. If a specific query path needs caching beyond what indexed Postgres provides, the answer is a Postgres materialised view, not a second cache vendor.
