# v1 ingestion stack and manual-override fallback

v1 ingestion is built on three sources: nflverse (`nflreadr` / `nfl_data_py`) for play-by-play, EPA, rosters, snap counts, schedules, depth charts, and historical weather; The Odds API for live game lines and player props; and Open-Meteo for upcoming-game weather forecasts (chosen over the NWS API because the schedule includes international games and NWS doesn't cover them). EPA is ingested from nflfastR's pre-computed values and used as-is in v1 — building a custom EPA model is a legitimate v2 research investigation but holds v1 features hostage to a methodology project that isn't on the critical path. Aligning v1 on the de facto community standard means published analyses on this data are directly replicable by other analysts in the same ecosystem (a portfolio multiplier).

Every table that has automated ingestion must also be writeable from a small admin UI behind auth, as a *documented fallback path* — not the primary path. The fallback exists because a prior MLB project lost data when a published source broke silently (an `IMPORTHTML` URL stopped working) and "wait for the pipeline to fix itself" was unworkable mid-season. Manual override is especially load-bearing for injury reports (nflverse's `load_injuries()` is reportedly less reliable than ideal) and for post-game weather overrides on outdoor games where forecast diverges from actuals. This does not conflict with ADR-0001's "no user-generated content" stance: manual entry curates *facts about the world* via an alternative ingestion path, it does not store research outputs (picks, notes, predictions).
