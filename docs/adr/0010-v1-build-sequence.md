# v1 build sequence: vertical slices, engine-first, research section in parallel

v1 is built as a sequence of **vertical slices**, each exercising every architectural layer (ingestion → schema → query → view → auth → deploy), rather than horizontal layers (schema first, then ingestion, then UI). Integration risk surfaces at the cheapest possible moment — within the first slice rather than at the end. A **pre-slice spike** confirms the highest-uncertainty assumption (parquet-in-Node against nflverse data) in isolation before the slice scaffolding commits to it — a few hours of local Node script, no Next.js setup, no schema. If the spike fails, ADR-0008's GitHub Actions contingency triggers before any schema, UI, or deployment work happens.

After a minimal first slice (gated Slate Dashboard skeleton reading hand-populated `teamWeekStats` with *realistic-looking* values — not arbitrary numbers, because UX problems are easier to see against plausible data), subsequent slices are sequenced **engine-first**: real weekly nflverse ingestion + EPA aggregation + in-house ELO computation; then player-level ingestion + denormalised opponent rank fields; then The Odds API ingestion. Only then does the second view (Game Detail Page) land, followed by Player Page, then Props and Team pages. The engine-first stance — versus building all five views against hand-populated data and layering ingestion later — avoids the hidden cost of encoding view-level assumptions about data shape that may not match what real ingestion produces. Views built against real data don't need a "revisit and revise" pass when ingestion finally lands.

The research section ships **in parallel with engine work, not queued behind the gated app**. The first publish happens at a natural break point (around Slice 3, when real data has been ingested enough to make charts that aren't synthetic), most likely a methodology piece — "here's how I'm computing EPA-based matchup edges, with examples from real ingested data." Reasoning: the research section is the portfolio surface (ADR-0005), and leaving it for last means portfolio value doesn't materialise until v1 is otherwise feature-complete — the wrong order of operations for a project framed as a portfolio piece. Publishing once also derisks the MDX pipeline (snapshot mechanism, live-query chart components, SEO/indexing) at a moment when pausing engine work for a few hours is cheap. Additional publishes follow at natural break points as the project produces material worth writing about.

Historical backfill (the local Python script from ADR-0008) is **not on the v1 critical path**. It runs once when current-season ingestion is stable and historical data is needed for splits or investigations — most naturally between The Odds API slice and the Game Detail Page slice. The slice numbering and ordering above is illustrative of the principles, not load-bearing; the durable calls are: vertical slices, pre-slice spike for the highest-risk assumption, engine-first after the first slice, research section in parallel, backfill off the critical path.

## Engine-work split (post-Slice-3-grilling)

The original "engine-first after the first slice" framing collapsed weekly nflverse ingestion, player-level ingestion, and The Odds API into one engine block. Slice 3 grilling on 2026-06-11 split that block into three sequential slices on **error-budget-management grounds**: bundling concurrent novel mechanisms (parquet streaming, jobQueue, MOV-ELO, player denormalisations, odds polling) into a single slice produces failure modes that are hard to attribute when multiple new systems break simultaneously — narrow scope keeps the candidate set small when something breaks.

Updated slice numbering: **Slice 3** team-level ingestion + MOV-ELO (the Phase 3a backfill plus the Phase 3b weekly cron writing `game`, `drive`, `play`, `teamWeekStats` — and the bundled MOV-ELO methodology publication per the Slice 3 ship criteria); **Slice 4** player-level ingestion + denormalised opponent rank fields (lights up the Player Page); **Slice 5** The Odds API ingestion (lights up the line columns on the Slate Dashboard and Game Detail Page); **Slices 6-9** the page slices (Game Detail, Player, Props, Team + Team Leaderboard). The principles (vertical slices, engine-first, research in parallel, backfill off the critical path) are unchanged; the numbering and the engine-block decomposition are the only edits.

## Update 2026-06-18: MOV-ELO methodology publication cut from Slice 3

The **bundled MOV-ELO methodology publication** referenced above (`/research/elo-methodology`, snapshot-mode per ADR-0007 — Slice-3 decision #11; also the "most likely a methodology piece" first-publish in the parallel-research paragraph) is **cut**: not written, and not replaced by a user-facing explainer. Rationale: the ELO is user-facing infrastructure, not a portfolio piece; the target audience already understands ELO; and the calculation is exhaustively documented in ADRs 0014 / 0021 / 0022, which stand as the methodology documentation in the piece's place.

**Consequence:** Phase 3b is the **only remaining Slice-3 sub-slice** — Slice 3 completes when Phase 3b ships. The "research section ships in parallel" *principle* above is unchanged; only this specific first-publish vehicle is cut, so the next research-section publish moves to a later, as-yet-unchosen break point.

This removes the planned vehicle for **ADR-0012 ship criterion #2** ("one substantive published investigation"). That criterion was **deferred to post-v1** (resolved 2026-06-18) — see [ADR-0012](0012-v1-ship-criteria-and-scope.md) and [ADR-0025](0025-v1-is-the-working-tool-research-deferred.md), which records the load-bearing upstream reasoning behind both this cut and the deferral: **v1 is a working tool for the friend group; the research/portfolio dimension is the post-v1 sandbox that grows out of it, not a gate on shipping it.**
