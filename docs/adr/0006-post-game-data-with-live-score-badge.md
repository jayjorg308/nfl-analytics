# Post-game data with live-score badge

v1 is a post-game data system: stats, EPA, ELO updates, hit-rate refreshes, and line history land after games finalize. The schema, ingestion stack, and UI all assume completed games as the unit of update — not in-progress drives. The load-bearing usage moment is Sunday-morning *prep* (not during-game tracking), and the pre-kickoff matchup analysis, props, and lines researched then are exactly what stays useful at 4pm ET. Considered a full-live posture (in-game EPA, live stats, websocket-driven UI) and a lazy-stats posture (live scores + completed-game stats only); rejected both because the real-time data plumbing would compete with the analytical work that should headline the project, and friends watching games already have broadcasts, RedZone, and their sportsbook apps for live state.

To prevent the "looks abandoned" failure mode on Sunday afternoons (a dashboard at 4pm showing "kickoff in 3 hours" for a game actually in Q3), in-progress game cards render a minimal **live-score badge** — current score + quarter/clock only, nothing else. No live EPA, no live stats, no live line movement. Data source is ESPN's hidden API (free, no credentials, community-standard for free live NFL scores), called server-side at page-load with ~30s caching and graceful fallback to "no badge" if the endpoint fails. The badge is purely informational, so the brittle/unofficial ESPN dependency is an accepted risk with a documented fallback path; nothing else in the system depends on it.

Post-game ingestion is triggered ~30 minutes after each day's last game ends — Sunday Night Football wrap triggers Sunday's run, Monday Night Football triggers Monday's, Thursday Night Football triggers Thursday's. The freshness contract: by Monday morning, every Sunday game's stats, EPA, ELO update, and hit-rate impact are fully integrated. Tuesday-morning freshness would break the weekly research workflow because Tuesday is when next week's slate prep begins, and that prep requires the previous week to be fully closed. A schedule-driven cron is the v1 mechanism; an event-driven trigger keyed off actual "game finalised" status is a viable v2 refinement.
