# Drive table and parquet-in-Node spike findings

A pre-Slice-1 spike validated parquet-in-Node ingestion against real nflverse data, confirming the ADR-0008 primary path and demoting the GitHub Actions / Python escape hatch from "likely contingency" to "documented fallback we don't expect to need." The spike used `hyparquet` against a current-season play-by-play release on a developer laptop, parsed the schema, materialised rows into TS types, and verified end-to-end that the column shapes, type quirks, and value ranges match what production ingestion needs to handle. Result: no friction substantial enough to trigger the contingency; the Node ingestion path stays as documented in ADR-0008.

The spike surfaced one schema-shaping decision: **drives become a top-level `drive` table rather than living as columns on `play` or as a JSON aggregate**. nflverse's parquet replicates the full drive context (drive number, possession team, play count, time of possession, start/end transitions, end yardline, result) on every play row inside that drive. Three modelling options were on the table — (a) leave drive context denormalised on every play row, matching the parquet shape; (b) extract a `drive` table during ingestion and reference it from `play.driveId`; (c) store drive metadata as JSON on the first play of each drive. (a) was rejected because every drive-level analytical query ("longest sustained drives this season," "drives starting after a turnover," "scoring rate by start field position") would have to deduplicate over `play` rows on read, and drive-level analytical queries are central to several v1 research investigations already named in the design — wrong-shape per ADR-0009. (c) was rejected because JSON columns trade query ergonomics for storage savings the data scale does not need. (b) is the chosen shape: drives extract to their own table during ingestion (deduplicate by `(gameId, fixed_drive)`), and `play.driveId` references the parent row.

The spike also confirmed several **field-mapping conventions** that any non-trivial Node ingestion code has to encode. Booleans in the parquet are stored as DOUBLE 0/1 (`pass`, `rush`, `complete_pass`, `interception`, `success`, `drive_inside20`, `drive_ended_with_score`) and need explicit conversion at the type boundary. Dates and times are BYTE_ARRAY strings — `game_date` parses as `YYYY-MM-DD`, `time` as `MM:SS` within quarter, `time_of_day` as ISO timestamp — none auto-convert through hyparquet's typed accessors. Score state has three distinct representations: `home_score`/`away_score` is the *final* score replicated on every row, `total_home_score`/`total_away_score` is the in-game score at the play, and `posteam_score`/`defteam_score` is the in-game score from the possession-team perspective. v1 stores `posteam_score`/`defteam_score` on `play.scoreOffense`/`scoreDefense` because the analytical use ("how does behaviour change when trailing by 7 vs leading by 7") naturally reads from the possession team's vantage. Weather is partially denormalised in the source — a free-text `weather` string alongside structured `temp` / `wind` integer columns (always null for domes); ingestion uses the structured columns when populated and ignores the free-text version. `fixed_drive` is the canonical drive number; the raw `drive` column is preserved by nflverse for compatibility but `fixed_drive` is the field that respects mid-game corrections and is what `drive.driveNumber` reads from.

Implementation note: detailed parquet → Postgres field mappings live in `docs/parquet-mapping.md`; ingestion sequencing (which tables update in what order after games complete) lives in `docs/schema-design.md`. This ADR captures the decisions that justify their shape, not the mappings themselves.
